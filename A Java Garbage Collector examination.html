A Java Garbage Collector examination

Published 5 months ago — 12 min read
By Nguyễn Khôi Nguyên

In software development, performance concerns have always been one of the top topics in discussions. Not only because it's vital to the system itself, but it also directly affects end-user experience as well.  

In this topic today, we focus on java applications and GC (garbage collector) tuning, as a way to make small and meaningful changes that can affect the system in big ways under certain circumstances.

What is GC?

GC or Garbage Collector is a form of automatic memory management in Java.  

It allocates from and gives back memory to the operating system. The allocated memory will be given to our application when it is requested. The garbage collectors then determine whether the memory is still used after an amount of time and reclaim if necessary.

Everything above is done inside Heap, a space for dynamic memory allocation on runtime to contain all java Objects. Along with Heap, there is Stack that contains local variables and function call that support for thread executions.

GCs come with the JVM included in the JRE (or JDK) that we use for our java application.

How do these GCs collect back unused memory?

Naively and purely theoretically, it would be to scan over all of the heap to see which object is live (referenced) and which is dead (not referenced to). All dead objects can then be cleared out and their memory returned to the system.

Obviously, the method above is slow and cumbersome so GCs don't actually do that. Instead it leverages generational collection, a method based on the Young/Weak Generational Hypothesis which states that most objects allocated don't get to live long and would survive for only a short period of time.

Take figure A.1 (courtesy of oracle) for example:

The x axis shows object lifetime and the y axis shows the bytes surviving after the collections.

The sharp peak at the top left indicate objects that live shortly, typically objects that are created in a loop after we exit the loop. Some objects do live longer so the line stretches to the right side, instance objects, objects in static variables that live to the end of the application life cycle.

Based on this finding, the GC divides the heap into contiguous regions called generations. Each indicate the time they have lived in the system. After a region is full, a sweep called Garbage Collection will be invoked to scan the objects and move them to their right memory space.

The method

Initially, all objects stay in the Eden space and 1 Survivor space is free at any moment.  

After a garbage collection is triggered, all live objects in Eden space can be allocated to the free Survivor space. Any live object in the other Survivor space will be reallocated to this Survivor space also. After a certain number of reallocations between the survivor spaces, objects that still live will be moved to the Old space.

When a garbage collection is triggered, all other threads in our java application will be suspended. The event is called "Stop the World" event.

To measure performance of these garbage collections we should be concerned with:

Throughput: the percentage of total time spent not in GCs.

Latency: the responsiveness of the program.

Choosing the right size for each generation space is important.  

For example, maximizing the Eden space will increase throughput. But each pause would take more time and contribute greatly to the overall latency of the application.  

Doing the opposite by rising the Old space will lower latency of each collection with a decrease in throughput as well.

There is no right way to choose the right size of a generation. This would depend on your application's characteristics and needs.

Do we really need to care about GC in all our apps?

No, not really, not all of the time.

For some apps, GC don't matter much, as long as there's still space left for the programs to run.

In contrary however, for enterprise-class applications which involve large amount of data going in and out the system, multi-threads and high request frequency rates, if we don't optimize and tune the GC, those small pauses may collectively contribute to a noticeable loss of performance.

Take this example from oracle official guidelines

Assuming we have a perfectly optimized application, save for only the GC, which runs in single thread.

Starting with a uniprocessor, the program spends 1% CPU time doing garbage collection. By scaling up the system vertically by increasing to 32 processors, the system's parallelism, which is supposed to increase the overall throughput, instead is hindered by the synchronous garbage collection mechanism. As a result, nearly 75% throughput is lost.

So, it's safe to say, for some cases, GC tuning is necessary to maintain consistent application's performance.

What types of GC are there?

We already know the basic mechanism of how GCs collect unused memory space, but there are many available concrete algorithms to go about that and many implementations to choose from when you start up your application.

Below are the default Garbage Collectors for each java iteration starting from java 7.

Java Version Default Garbage Collector
1.7 Client mode: Serial Collector
Server mode: Parallel Collector
1.8 Client mode: Serial Collector
Server mode: Parallel Collector
1.9 G1 Garbage Collector

Serial Collector

Does everything within 1 thread, this collector is suited for machines with single processor.

Can be enabled with the option -XX:+UseSerialGC

Uses a compact collection method, old memory will be put as at the beginning of the heap so new memory allocation can be contiguous chunks at the end of the heap making the allocation faster.

Used mainly on single-threaded client style machines without strict pause time requirements

Parallel Collector

Is also known as the Throughput Collector

Uses multiple threads to speed up the garbage collection. For a system with <N> physical hardware threads, a fixed fraction of <N> will be used for collector thread size.

By focusing on the throughput, maximum latency may rise to a high level occasionally during the major collection.

Suitable for applications that has to do multiple jobs at the same time, while maximum latency is not important (multi-threaded computing, report generation for big data set,...)

Can be enabled using options:

  • -XX:+UseParallelGC for multi thread young generation collectior, single thread old generation collector
  • -XX:+UseParallelOldGC for both multithread on old and young generation collectors

At the start of each collection, each thread would register a space inside the old generation. Thus, by doing so, these spaces would cause a fragmentation effect as time goes. The effect may be reduced by decreasing the number of collector threads.

How to tune: in the order below

Pause time: -XX:MaxGCPauseMillis=<N> if maximum pause time cannot be achieved, the generations are shrunken in turn

Throughput: -XX:GCTimeRatio=<N>, if not achieved, the generations at faults are increased in turn as well

Footprint:

-Xmx<N>, depicts maximum heap size allowed

-Xms<N>, initial heap size.

Default heap size: maximum is 1/4 of total memory, initial size is 1/64 of total memory.

Mostly Concurrent Collectors

Do parts of their work concurrently

HotSpot VM has 2:  

  • Concurrent Mark Sweep (CMS) collector
  • Garbage-First (G1) collector

Run concurrently with application threads, takes processor resources to do garbage collecting thus resulting in shorter major collection pause time, though throughput is a bit lower than other collectors.

Concurrent Mark Sweep Collector (CMS Collector)

Preferably for applications with large datasets that live long (big old generation space) with a priority for short pauses like graphical applications, servers responding to requests,...

It uses separate GC threads to scan the old for live objects to reduce major pauses.

Does not compact live object after each sweep so when fragmentation happens, allocates a bigger heap

Is enable by: -XX:+UseConcMarkSweepGC

When the collector fails to allocate a free space in the old generation, a major pause happens with all other application threads stopped. This is called concurrent mode failure. When this happens, it indicates that we need to tune the collector parameters

To debug: -Xlog:gc

Garbage-First Collector

Has been available since java 7, but not until java 9 that it becomes the default GC

Suitable for multi-processor systems with lots of physical memory.

Aims for best balance between throughput and latency

Heap size support larger than 4GBs

On the other hand, doesn't do well with small heaps.

This is created to replace the CMS Collector and become the default Garbage Collector, in contrast to CMS Collector, this is a compact Collector.

Can be enabled by: -XX:+UseG1GC

Heap is a contiguous space divided into large fixed size regions, about 2MB each by default.

Each region can be Eden space, Survivor space, Old generation or unassigned.

By using these non-contiguous regions, the G1 may pick a small group of region to collect and finish quickly instead of doing the whole space like the other collectors thus having the ability to support large heaps

The amount of live data in each region is tracked, and when a collection is triggered the G1GC will clear the ones with the most ‘garbage’ first hence the name.

Epsilon Collector

Epsilon is a "no-op" GC.

It does handle memory allocation but doesn't reclaim them back when objects are not live.

When your application uses up all the heap, the JVM would shut down.

Epsilon is for when you know exactly what your application does and want to squeeze every last bit of performance out of it.

Activate with: -XX:+UnlockExperimentalVMOptions  -XX:+UseEpsilonGC  -XmxXg

XmxXg is to set the heap size for your application, example: -Xmx4G for a heap of 4GB

Z Garbage Collector

This garbage collector is design for low latency with support to multi-terabytes heaps, even lower than G1 Collector default 200ms max pause time. Throughput is also guaranteed.

It was first introduced in java 11, until java 12 the garbage collector is still in experimental mode.

Does all expensive work concurrently which doesn't stop application thread for more than 10ms.

Only available on 64 bit linux, it uses the 64 bit pointers to encode additional information in each pointer.

Can be unlocked with: -XX:+UnlockExperimentalVMOptions -XX:+UseZGC

Shenandoah Garbage Collector

Developed at red hat and has been around for years. The garbage collector is now a part of java 12.

Trades CPU resources and memory space to achieve low pause time with its concurrent operations.

Also promises maximum pause time no longer than 10ms like the Z Garbage Collector

Can be unlocked with: -XX:+UnlockExperimentalVMOptions -XX:+UseShenanodoahC  

Shenandoah needs to collect heap faster than the application eats up, if it can't, a failure would occur and trigger 1 of the 3 configurable mechanism:

Pacing: if the garbage collector doesn't allocate fast enough, it starts up more threads to do the allocating, it it fails, comes the next mechanism.

Degenerated GC: when allocation failure occurs, it trigger a stop-the-world event. Usually this is when most of the collection cycle are nearly completed so this event should be brief. If this one fails, escalate to the next mechanism.

Full GC: This would almost guarantee 100% success rate for memory allocation unless there's no space left in the heap.

Shenandoah offers the same advantages like the ZGC with larger heaps. But it has been around for longer than ZGC so it has seen more testing and tuning.

There goes the introduction to all currently available Garbage Collectors from java 7 to java 12.

In this next section, we discuss in more details when to use what Garbage Collector.

Serial Garbage Collectors: you should use this collector if your application is a client type app that doesn't need multi-threading computing.

Parallel Garbage Collectors: this collector is also called the throughput collector for its use of multiple threads to do young generation collection. On the other hand, it may cause fragmentation in the old generation that would lead to long major collection. So use it if your application can take these occasional long pauses that can last up to 1 second. Suitable for utility applications that require maximum throughput to process data and do computation.

CMS Garbage Collector: this is a concurrent collector which focuses on reducing major pauses. It is most suitable for applications that doesn't tolerate long pauses like apps that require smooth graphical interfaces, servers serving requests that need to qualify a maximum latency constraint,... Though it would require more CPU resources to do its job and shouldn't be used for heaps bigger than 4GBs.

G1 Garbage Collector: this is made to replace the CMS Garbage Collector with enhancements to support larger heaps (up to 10 GBs or larger), balancing between throughput and latency. By using this, just like the CMS, be prepared to trade CPU time for better latency. Although the throughput for this collector may be slightly worse than other collectors. Suitable for server type applications that require large heaps, deployed on system with enough CPU to spare.

Epsilon Garbage Collector: This is a no-op garbage collector, which means it doesn't collect garbage at all. It just helps allocate memory for your application. Use this if and only if you know exactly what you are doing with your available memory and your application doesn't create garbage. On the other hand, by cutting out the garbage collecting mechanisms, your application achieves maximum performance possible.

Z Garbage Collector: support big heaps, multi-terabytes at that and in the same time, offer no more than 10ms maximum latency (works are being done to keep this to 1ms). The collector is NUMA-aware (which needs another article to dive into). So by enabling the NUMA-aware option on NUMA machines, the different in performance can be noticeable. This collector can be enabled on 64 bit linux machines only. Use it if your server application qualify for all the needs and requirements above.

Shenandoah Garbage Collector: this garbage collector is similar to the Z Garbage Collector. It is only available to java 12, OpenJDK. Oracle JDK doesn't support this GC. Suitable for large heap size only and if you don't have a pause time constraint, consider using G1 Garbage Collector instead.

In this last section, we will find out how to tune each Garbage Collector

Serial Garbage Collector: There's no tip but to keep your dataset approximately 100MB

Parallel Garbage Collector:

On a machine with <N> hardware threads, the default total GC thread is a fixed fraction of <N>

Option Desc Tips
XX:ParallelGCThreads Total number of GC threads more threads means more throughput. Too many would lead to defragmentation with longer major pauses
XX:MaxGCPauseMillis The maximum pause time desired decreasing this value may lead to lower throughput
XX:GCTimeRatio The desired throughput value indicate 1/(1+) time in GC. Example: -XX:GCTimeRatio=19 means the goal is 1(1+19) = 1/20 time in GC
Xms initial heap size The GC will start at the initial heap size and increase to the maximum heap size until the goals are met
Xmx max heap size

CMS Collector:

Unlike the serial or parallel collector that starts collecting from the old generation when it becomes full, the CMS Collector schedule to collect from the old generation before it is full with dynamic estimates.

Option Desc Tips
Xlog:gc*=debug print debug info
XX:CMSInitiatingOccupancyFraction the percentage of occupancy in the old generation to start a collection

G1 Collector:

the G1 collector is good enough on its own with the default configuration. But in certain cases, we can tune it to our need.When moving from other Garbage Collector to G1, remove all options concerning garbage collection, keeping only the maximum heap size.

Option Desc Tips
XX:MaxGCPauseMillis the desired pause time in milies increase this value if you need throughput, decrease it for lower latency
Xmx max heap size
Xlog:gc*=debug print debug info

Major collection would lead to long pauses when the GC allocate too many objects without reclaiming them in time. One of the reasons could be because of too many objects allocated before they can be reclaimed. Or the humongous objects are too many leading up to the need for collection.

The below options may help with this case:

Option Desc Tips
XX:G1HeapRegionSize the size of the heap region Increase this to make space for humongous objects. If there's no space available for the humongous object, a collection would be called
XX:ConcGCThreads number of concurrent GC threads increasing this speeds up the collecting process but eats up more CPU resources
XX:+DisableExplicitGC disable the explicit GC calls by the app unnecessary explicit GC calls by devs like System.gc() may lead to unwanted performance issues, this options disable that function call

To tune for latency, use the below options

Also, one should always check for logs to determine if the GC is getting enough CPU resources. If it lacks CPU time, latency would rise.

Option Desc Tips
XX:+AlwaysPreTouch allocate the heap regions on JVM startup allocating heap regions may take time with big heaps, pre-touch them on startup to potentially lower the response time

Epsilon Garbage Collector

The most important option for this collector would be the maximum heap size

Option Desc Tips
Xmx Maximum heap size

Z Garbage Collector

This is a concurrent garbage collector, so we need to leave sufficient heap space for the live objects and spare space to do heap allocation when the application threads are running.  

Also, we can define the total threads used for the GC, too many would eat up our application CPU time, too few would lead to the application allocating memory for objects faster than the GC can reclaim.

Option Desc Tips
Xmx Maximum heap size remember to leave enough to save room for object reallocationing
XX:ConcGCThreads total number of GC threads

In conclusion of this post, GC tuning is sometimes necessary and sometimes isn't.

It is our job to know and understand the nature of JVM's GC mechanism to ensure maximum performance for our java applications.

And hopefully with this post, may you find the most suitable GC for your need .!
