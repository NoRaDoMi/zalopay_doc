Sử dụng Git Rebase một cách hiệu quả

Bài viết này chia sẻ cách sử dụng git rebase, một trong những công cụ mạnh mẽ mà Git cung cấp, để giải quyết các vấn đề thường ngày. Git Rebase đã giúp cuộc sống của tác giả dễ dàng hơn, git graph thẳng thớm hơn, qua đó giúp kiểm soát tốt hơn lịch sử của code base.

Published 3 months ago — 6 min read
By Lê Thái Phúc Quang

Bài viết này chia sẻ cách sử dụng git rebase, một trong những công cụ mạnh mẽ mà Git cung cấp, để giải quyết các vấn đề thường ngày. Git Rebase đã giúp cuộc sống của tác giả dễ dàng hơn, git graph thẳng thớm hơn, qua đó giúp kiểm soát tốt hơn lịch sử của code base.

Tác giả giả định bạn đã biết và đã từng xài Git, cũng như hiểu được một vài khái niệm cơ bản trong Git (commit, branch, remote,...).

Cú pháp

Cú pháp tổng quát của git rebase như sau:

git rebase --onto <new_base> <current_base> <branch_to_rebase>

Câu lệnh sẽ được diễn dịch như sau: "Rebase branch_to_rebase lên trên new_base, nhưng chỉ những commit không nằm trong current_base".

Giá trị mặc định:

  • new_base là tham số bắt buộc, lúc nào cũng phải có
  • current_base là giá trị của new_base
  • branch_to_rebase là branch hiện tại

Để dễ nhớ khi diễn dịch, chúng ta nên có thói quen checkout ra branch cần rebase trước khi thực hiện rebase:

git checkout <branch_to_rebase>

Như vậy, cú pháp bây giờ chỉ còn lại:

git rebase --onto <new_base> <current_base>

Diễn dịch: "Rebase branch hiện tại lên trên new_base, nhưng chỉ những commit không nằm trong current_base".

Ngoài ra, new_base và current_base không nhất thiết phải là branch, mà có thể là một ref bất kì: tag, commit hash, ...

Rebase một tham số

Ở phiên bản đơn giản nhất, câu lệnh git rebase chỉ cần một tham số:

git rebase <new_base>

Câu này tương đương với câu lệnh đầy đủ:

git rebase --onto <new_base> <new_base>

Diễn dịch: "Rebase branch hiện tại lên trên new_base, nhưng chỉ những commit không nằm trong new_base".

Hơi rắc rối phải không, đừng lo, hãy xem qua một vài ví dụ.

Ví dụ

Rebase rẽ nhánh

Trăm nghe không bằng một thấy:

A--B--C--D--E   [master]
    \
      F--G--H   [feature]

Hiện tại, branch master và branch feature đang rẽ theo 2 đường khác nhau. Nếu ta checkout feature và rebase lên trên master:

git checkout feature
git rebase master

Graph sẽ trở thành như sau:

A--B--C--D--E               [master]
             \
               F'--G'--H'   [feature]

Như vậy, Git đã thực hiện đúng như câu lệnh:

git rebase --onto master master

"Rebase branch hiện tại lên trên master, nhưng chỉ những commit không nằm trong master (F, G, H)".

Lúc này thì feature và master đã thẳng hàng.

Rebase thẳng hàng

Nếu ta tiếp tục checkout master và rebase lên trên feature:

git checkout master
git rebase feature

Git sẽ lại tiếp tục thực hiện:

git rebase --onto feature feature

"Rebase branch hiện tại lên trên feature, nhưng chỉ những commit không nằm trong feature (không có cái nào cả)".

Bởi vì không có commit nào thỏa yêu cầu, nên Git chỉ đơn giản trỏ master đến cùng commit với commit feature đang trỏ. Kết quả:

A--B--C--D--E
             \
               F'--G'--H'   [master][feature]

Xử lý conflict

Khi thực hiện rebase, các commit sẽ được đưa lên trên đầu new_base từng commit một. Trong quá trình đó, nếu commit nào bị conflict, thì chúng ta sẽ phải giải quyết conflict, sau đó mới được rebase tiếp.

Trong ví dụ trên, khi rebase feature lên trên master, nếu commit F có conflict, chúng ta sẽ giải quyết conflict của F với master, sau đó mới đến lượt G, rồi cuối cùng mới đến H.

Rebase hai tham số

Nhắc lại (đỡ mất công bạn cuộn lên cuộn xuống), khi ta gõ lệnh:

git rebase --onto <new_base> <current_base>

Git sẽ "rebase branch hiện tại lên trên new_base, nhưng chỉ những commit không nằm trong current_base".

Bình thường, nếu git rebase chỉ nhận một tham số, thì cũng chỉ thường thường bậc trung như bao câu lệnh khác. Nhưng khi có sự xuất hiện của tham số --onto, Git Rebase lúc này sẽ trở nên tiện lợi hơn bao giờ hết.

Đi luôn vào ví dụ

            M--N--O   [new_feature]
          /
A--B--C--D--E         [master]
    \
      F--G--H         [feature]

Bây giờ, ngoài branch feature, chúng ta có thêm branch new_feature chứa một tính năng mới mà chúng ta cần.

Nếu chúng ta chỉ đơn giản rebase new_feature lên trên feature:

git checkout new_feature
git rebase feature

Chúng ta sẽ có graph:

A--B--C--D--E                       [master]
    \
      F--G--H                       [feature]
             \
               C'--D'--M'--N'--O'   [new_feature]

Rõ ràng là chưa đúng, vì chúng ta chỉ cần mang các commit M, N, O của new_feature vào feature, nhưng lúc này lại "dính" thêm các commit C, D của master nữa. (Tại sao lại thế? Bạn trả lời thử nhé).

Bây giờ xé nháp, reset lại lúc chưa rebase:

            M--N--O   [new_feature]
          /
A--B--C--D--E         [master]
    \
      F--G--H         [feature]

Để mang new_feature vào feature, ta thực hiện:

git checkout new_feature
git rebase --onto feature master

Khi đó Git sẽ "rebase branch hiện tại lên trên feature, nhưng chỉ những commit không nằm trong master (M, N, O)".

Graph lúc này trở thành:

A--B--C--D--E               [master]
    \
      F--G--H               [feature]
             \
               M'--N'--O'   [new_feature]

Nhưng đến đây, feature đã có những tính năng của new_feature chưa ạ? Xin thưa là chưa, chúng ta cần thêm một động tác nữa:

git checkout feature
git rebase new_feature

Khi đó, feature mới có được những tính năng của new_feature

A--B--C--D--E               [master]
    \
      F--G--H               
             \
               M'--N'--O'   [feature][new_feature]

Rebase với commit lặp

Phần này các bạn chỉ cần đọc cho biết, không có gì cần ghi nhớ cả. Giả sử chúng ta có graph:

A--B--C--D--E            [master]
    \
      F--G--H            [feature]
             \
               M--N--O   [new_feature]

Khi ta rebase feature lên trên master:

git checkout feature
git rebase master

Graph sẽ trở thành:

               F'--G'--H'   [feature]
             /
A--B--C--D--E               [master]
    \
      F--G--H--M--N--O      [new_feature]

Lúc này, feature đã có những commit mới nhất của master. Nhưng sau đó, chúng ta cần mang cả new_feature lên trên feature nữa.

Theo những gì đã trao đổi nãy giờ, chúng ta có thể thực hiện như sau:

git checkout new_feature
git rebase --onto feature HEAD~3

Nhưng nếu chúng ta muốn thử vận may, chúng ta có thể chỉ cần đơn giản:

git checkout new_feature
git rebase feature

Khi đó, Git sẽ mang cả 6 commit của new_feature (F ~ O) lên trên feature, nhưng sẽ tự động bỏ qua các commit có changes giống nhau, bằng cách so sánh patch-id của các commit.

Ở đây, nếu F có changes giống F', G giống G', H giống H', thì cuối cùng chỉ có 3 commit M, N, O được mang lên trên feature:

                           M'--N'--O'   [new_feature]
                         /
               F'--G'--H'               [feature]
             /
A--B--C--D--E                           [master]

Nhưng nếu không may mắn?

Trong trường hợp không may, các cặp F-F' G-G' H-H' có changes khác nhau (xảy ra khi rebase feature lên trên master trước đó), thì khi rebase new_feature lên trên feature, bạn sẽ phải một lần nữa giải quyết các conflict của F, G, H.

Kết

Bất cứ gì cũng vậy, nếu chúng ta muốn sử dụng những thứ mạnh mẽ, thì đều cần phải hiểu nó thật rõ.

Bài viết đã giới thiệu cho các bạn phần nào khả năng của Git Rebase, rebase với một tham số và hai tham số, rebase khi có commit lặp.

Git Rebase đã giúp cuộc sống của mình dễ dàng hơn, git tree của code base thẳng thớm hơn. Hy vọng bạn cũng đúc kết được những điều đáng nhớ cho riêng mình.

Chúc các bạn may mắn, YOLO!
